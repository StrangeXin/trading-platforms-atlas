# Prop Firm Operator — Prototype Build Order

Date: 2026-04-24
Status: v0-control

## 核心判断

如果真的开始做原型，最大的风险不是做不出来，

而是：

**太早把时间花在“像完整产品的东西”上，而不是先证明这台机器真的会咬人。**

所以 build order 必须围绕一个原则：

先验证毒性循环成立，再给它皮肤，再给它内容量。

## 原型开发的唯一总目标

在最短时间内验证这 4 件事：

1. 玩家能否快速理解 fantasy
2. 五个旋钮是否足以制造真实张力
3. 延迟后果是否真的可感知
4. 一局结束后玩家是否会想重开

如果这 4 件事里有两件不成立，立即停止扩张内容。

## 推荐的 7 个阶段

### 阶段 0：Paper Sim

#### 产出

- 手工跑表
- 资源更新规则草稿
- 10–14 周纸面模拟

#### 目标

先验证五个旋钮与六个资源有没有明显的脏张力。

#### 判断标准

- 是否很快出现“最优解”
- 是否真的能出现三种不同死法
- 是否能让坏决定短期变好看

#### 为什么先做这个

因为如果纸面都不好玩，代码只会把不好玩做得更贵。

### 阶段 1：Bare Weekly Loop

#### 产出

- 没有 fancy UI 的最小周循环
- 每周输入五个旋钮
- 输出六个资源变化

#### 目标

先做出：
- 选择
- 结算
- 下一周

不要事件，不要 narrative，不要结局。

#### 判断标准

- 玩家是否 2–3 周就明白自己在经营什么
- 有没有至少两种明显不同的操作风格

### 阶段 2：Delayed Consequence Layer

#### 产出

- `Complaint Echo`
- `Winner Visibility`
- 至少 2 周以上的延迟后果

#### 目标

这是最关键的一层。

如果没有延迟，这产品会退化成普通调参游戏。

#### 判断标准

- 第 1 周的决定能否在第 4–6 周回来找你
- 玩家是否开始觉得“我在吃未来”

### 阶段 3：Week Digest

#### 产出

- 每周结算页
- 大数字 + narrative feed + 风险提示

#### 目标

让系统开始“像产品”。

这里不是为了美观，而是为了把抽象变量翻译成可读经验。

#### 判断标准

- 玩家是否能说出“这周到底发生了什么”
- 玩家是否记得一周故事，而不只记得数值

### 阶段 4：Event Layer

#### 产出

- 先接 4–6 张关键事件卡
- 事件能读到当前状态并改变未来

#### 目标

把系统从“会算”推进到“会逼你表态”。

#### 判断标准

- 玩家是否真的会犹豫
- 是否存在“明知不好但还是按”的按钮

### 阶段 5：First Ending Pass

#### 产出

- 至少 1 个脏成功结局
- 至少 2 个失败结局

#### 目标

把一局做闭环。

#### 判断标准

- 玩家结束后是否会讲自己的死法
- 玩家是否会立刻想重开试另一种路线

### 阶段 6：Presentation Polish

#### 产出

- 控制室排版
- 状态词
- 标题语气
- 基础动效

#### 目标

直到这一步，才开始把它做得更“像成品”。

#### 判断标准

- 界面是否强化了权力感与发臭感
- 有没有开始抢夺系统本身的注意力

## 第一版最不该优先做的 6 件事

### 1. 不要先做多模板

先把一条线做毒。

### 2. 不要先做复杂 UI

先让机器成立，再给机器外壳。

### 3. 不要先做 meta progression

先确保单局本身能咬人。

### 4. 不要先做一堆事件

先证明 4–6 张就足够制造紧张。

### 5. 不要先做精细经济数值

先用归一化资源调出张力。

### 6. 不要先做完整法域/支付模拟

先把“监管和支付会卡你”这个感觉做出来。

## 推荐的实际实现顺序

如果真的开始做，建议按下面顺序推进：

### Milestone A：一个能跑的毒循环

包括：
- 五个旋钮
- 六个资源
- 10 周循环
- 没有事件

成功标志：
- 玩家已经能感觉到“增长很好看，但有代价”

### Milestone B：一个会记仇的系统

包括：
- 延迟后果
- 投诉回声
- 赢家可见度

成功标志：
- 玩家开始因自己前面的选择而后悔

### Milestone C：一个会说话的原型

包括：
- Week Digest
- Narrative feed
- 风险短句

成功标志：
- 玩家开始记住故事，不只记住数值

### Milestone D：一个会逼你脏下去的原型

包括：
- 4–6 张关键事件
- 事件选项和延迟反噬

成功标志：
- 玩家开始说“我知道这不对，但我还是点了”

### Milestone E：一个能讲死法的原型

包括：
- 脏成功
- payout 崩盘
- trust 崩盘
- 监管冻结

成功标志：
- 玩家结束后能复述“我是怎么死的 / 我最后变成了什么”

## 每个里程碑的 kill criteria

如果下面情况出现，应暂停扩张并回头修系统。

### Kill 1

玩家始终认为所有旋钮只是：
- “一个涨，一个跌”

而感受不到“在经营人性”

### Kill 2

玩家觉得最优路线太明显

### Kill 3

玩家不知道为什么会输

### Kill 4

玩家觉得自己只是被随机事件整死

### Kill 5

玩家对第一局没有故事记忆

## 最适合先做的测试方式

### 测试 1：设计者自跑

目的：
- 快速发现显性最优解

### 测试 2：懂经营游戏的人试玩

目的：
- 看他们会不会立刻破解系统

### 测试 3：不懂金融但懂游戏的人试玩

目的：
- 验证 fantasy 是否足够清晰

### 测试 4：只看 20 分钟首局试玩

目的：
- 检查第一次赚钱、第一次坏决定、第一次反噬是否成立

## 原型实现时最值得坚持的创意原则

### 1. 先做“毒性”，再做“复杂度”

复杂不等于有效。

### 2. 先做“记忆点”，再做“内容量”

玩家会记住：
- 第一次大促
- 第一次 viral payout
- 第一次监管来信

不会记住第 17 个细小参数。

### 3. 先做“反噬”，再做“扩展”

如果系统不会记仇，这产品就没有灵魂。

## 如果只允许做一个最小闭环

建议是：

1. 调五个旋钮
2. 跑一周
3. 读 Digest
4. 接一个事件
5. 再跑三周
6. 被第一次 payout / trust / regulator 问题打脸

只要这个闭环成立，整个项目就活了。

## 一句话总结

Prototype Build Order 的核心，不是排开发任务，

而是保护这款产品最重要的东西不要在实现过程中被稀释掉：

**先做出一台会诱惑你、会记仇、会收账的机器，再去把它做漂亮。**
